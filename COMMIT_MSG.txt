Sprint 6 (chunks 1-4) · Equipos · Organization members con invitacion por email

Permite que un constructor o tecnico invite a jefes de obra u otros miembros
para que puedan operar el dia a dia sin tener que pasar KYC ni desplazar al
dueno a cada obra. La cadena de custodia forense se mantiene: cada upload de
evidencia queda firmado por el user real que lo hace.

CHUNK 1 · Schema SQL Organizations + Members

  - Extiende tabla `organizations` existente (KYB de empresa) con:
      * owner_user_id (vinculo al user dueno)
      * description, org_type ('constructor'|'tecnico'|'promotor'|'mixed')
  - Nueva tabla `organization_members`:
      * role: 'owner' | 'member'
      * state: 'invited' | 'active' | 'revoked'
      * can_view_economics (toggle de visibilidad financiera, default false)
      * invitation_token (UUID para link de aceptacion)
  - Trigger auto-crea miembro owner al crear org
  - Trigger valida unico owner activo por org
  - Helper fn_is_org_active_member para RLS
  - RLS conservadora: miembros activos leen, admins escriben via RPCs

CHUNK 2 · 7 RPCs de gestion

  sf_create_organization        · crear org (caller queda owner)
  sf_invite_org_member          · owner invita por email
  sf_accept_org_invite          · valida que el email del user coincide con
                                  el invitado, activa la membresia
  sf_revoke_org_member          · DELETE fisico si era invited, soft delete
                                  si era active (preserva trazabilidad)
  sf_update_member_permissions  · owner cambia can_view_economics
  sf_list_my_orgs               · lista mis orgs (owner o miembro) con stats
  sf_get_org_members            · lista miembros, owner ve todos

CHUNK 3 · Pantalla "Mi equipo" (Flutter)

  - lib/features/organization/data/organization.dart           (modelos)
  - lib/features/organization/data/organization_actions.dart   (service)
  - lib/features/organization/data/organization_providers.dart (Riverpod)
  - lib/features/organization/presentation/pages/my_team_page.dart
  - lib/features/organization/presentation/sheets/org_action_sheets.dart
      (4 sheets: crear org / invitar / revocar / cambiar permisos)
  - Tile "Mi equipo" en perfil + ruta /profile/team
  - Card de org con badge OWNER, chips de miembros, CTA invitar
  - Lista de miembros con avatares + pills (rol, estado, ve economicos, TU)
  - Empty state amigable cuando aun no hay organizacion

CHUNK 4 · Email transaccional + landing de aceptacion

  - Edge Function send-org-invite (Deno + Resend)
      * Lee miembro + org + datos del owner
      * Construye HTML con branding PactStream (header navy, CTA azul)
      * Envia via Resend al email invitado
      * URL del link: /#/org-invite?token=xxx (hash routing de Flutter)
  - Cliente Flutter: inviteMember() invoca la Edge Function tras crear
    la fila. Best-effort: si el email falla, la invitacion queda en BD.
  - lib/features/organization/presentation/pages/accept_org_invite_page.dart
      * Procesa el token al cargar
      * Si no hay sesion: redirige a login conservando el token
      * Si hay sesion: llama sf_accept_org_invite y muestra exito/error
  - Ruta /org-invite anadida al router (publica, no requiere sesion)

PENDIENTES IMPORTANTES (Sprint polish)

  - Registro simplificado para invitados (saltar rol/KYC/empresa si
    vienen con invitation_token)
  - Path-based routing en Flutter para que el link del email no necesite
    /#/ (estetica)
  - CIF obligatorio o auto-generado en creacion de org (constraint
    NOT NULL UNIQUE heredado)

COMPATIBILIDAD

Las tablas y RPCs son aditivas: nada se rompe en sprints anteriores.
La tabla `organizations` mantiene todos los campos KYB (cif, kyb_status,
mangopay_user_id, etc.) y solo se le anaden 3 columnas nuevas.
